# MAC MVP Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the first-phase MAC MVP: local task handoff, context handoff, SQLite task ledger, risk-based verification contract, and CLI bridge.

**Architecture:** Keep one domain/service core shared by in-process API, HTTP/WebSocket, and CLI. SQLite Task Ledger is the first-phase source of truth. Context is passed through ContextBundle references; MCP remains the resource/tool/context layer, while MAC owns scheduling, ledger, handoff, and quality gate decisions.

**Tech Stack:** Python 3.11+, Pydantic, SQLite, Typer or argparse, pytest, FastAPI/WebSocket if HTTP mode is implemented in the same milestone.

---

## Scope

First phase includes:
- HTTP/WebSocket or in-process API
- SQLite Task Ledger
- AgentCard and A2A-compatible task profile
- ContextBundle
- Risk-Based TestContract
- CLI bridge
- TTL, max_hops, CAS, audit trail, quality gate

First phase excludes:
- gRPC and Protobuf
- Redis Pub/Sub
- PostgreSQL
- Cloud Bridge
- Hybrid Bridge
- automatic capability verification

## Acceptance Criteria

The MVP is accepted when:
- A task can be submitted, handed off, started, quality-checked, completed, and queried from SQLite.
- Every state transition writes an audit entry with `trace_id`.
- Context is passed through ContextBundle summary and references, not large raw payloads.
- High-risk tasks cannot complete without passing TestContract evidence.
- CLI bridge can register an Agent, submit a task, accept it, complete it, and print audit history.
- No Phase 1 code depends on gRPC, Redis, PostgreSQL, Cloud Bridge, or Hybrid Bridge.
- All verification commands pass.
